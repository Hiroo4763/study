# 2주차 - 암호 기초

## 2.1 개요

## 고전 암호

모두 대칭키 암호(암호화, 복호화 모두 같은 키)다.

### 단순 치환 암호

현재 문자를 n번째 앞에 있는 무자와 서로 치환해서 암호화하는 것

**시저 암호**: 단순하게 암호화하고 복호화하는 방법

### 이중 전위 암호

평문을 주어진 크기의 행렬 형태로 배열하고 행과 열을 기술된 순서에 따라 바꿈

메세지에 나타나는 문자를 감추치 않음

통계적 정보에 의존하는 공격은 막을 수 있음

### 일회성 암호

**암호 체계 중에서도 가장 안전**

- 버넘 암호라고도 불림
- 안정성이 증명된 암호체계
- 메시지와 동일한 길이의 무작위 비트열로 구성된 키가 필요

XOR 방식을 사용하기 때문에 깊이 있게 사용되면 키가 사라짐 
ex) C1 ⊕ C2 = P1 ⊕ K ⊕ P2 ⊕ K = P1 ⊕ P2


### 코드북 암호

코드북 암호는 단어와 해당 단어에 할당된 코드가 수록된 사전과 같음

*코드북 잃어버리면 안된다... 약간 군대 CEOI 같은 느낌...*

**덧셈**: 기존 코드북을 오래 사용하는 방법, 암호화-복호화 할때 덧셈북, 코드북 모두 필요

## 암호 분석의 분류

- 목표
평문과 키 두 가지를 모두 찾아내는 것

- 알려진 평문
힌트가 있는 상태? 공격에 성공할 가능성은 높음

- 선택된 평문
실험 가능한 상태? 공격자에게 유리한 것은 적응적으로 선택된 평문 공격

**순방향 탐색 공격**
: 공개키 암호체계에 사용, 공격에 대비하여 평문 메시지 공간이 충분히 커야 함.

# 3주차 - 대칭키 암호

## 3.1 개요

대칭키 암호는 같은 키로 암·복호화하는 방식. 스트림과 블록 두 종류가 있음.

### 스트림 vs 블록

| 구분 | 스트림 암호 | 블록 암호 |
|---|---|---|
| 처리 | 비트 단위 | 블록 단위 |
| 원칙 | 혼돈만 | 혼돈 + 확산 |
| 예시 | A5/1, RC4 | DES, AES, TEA |
| 용도 | IoT, 실시간 통신 | 일반 데이터 (주류) |

### 섀넌의 2원칙

- 혼돈(Confusion): 키-암호문 관계를 복잡하게 → 치환으로 달성 (예: AES S-Box)
- 확산(Diffusion): 평문 1비트 변화 → 암호문 여러 비트 변화 → 전위로 달성 (예: ShiftRows)

* 전위: 위치를 바꾸는 것
---

## 3.2 스트림 암호

키 K를 긴 키스트림으로 늘린 뒤 평문과 XOR.

> C = P ⊕ keystream

### A5/1 (시프트 레지스터)

3개의 LFSR 사용, 총 64비트 = 키.

- X: 19비트, Y: 22비트, Z: 23비트
- **다수결 함수 Maj**로 m 계산 → m과 일치하는 레지스터만 시프트
ㄴMaj: 다수가 0이면 0을 반환, 아니면 1을 반환

### RC4 (SSL, WEP)

256바이트 검색표 S를 스스로 수정하는 바이트 기반 암호.

**초기화**

```
for i = 0 to 255
    S[i] = i
    K[i] = key[i mod N]
j = 0
for i = 0 to 255
    j = (j + S[i] + K[i]) mod 256
    swap(S[i], S[j])
```

**키스트림 생성**

```
i = (i + 1) mod 256
j = (j + S[i]) mod 256
swap(S[i], S[j])
t = (S[i] + S[j]) mod 256
keystreamByte = S[t]
```

> 8비트 최적화라 현재는 거의 안 씀.

### 스트림 → 블록 이동 이유

시프트 레지스터는 하드웨어엔 빠르지만 소프트웨어엔 느림. 요즘은 소프트웨어 처리라 블록이 주류.

---

## 3.3 블록 암호

### 파이스텔 구조

- F의 역함수 불필요 → 복호화에도 F 그대로 사용
- 보안성은 F와 키 스케줄이 결정
- 단점: 매 회전마다 비트 절반만 영향 → 라운드를 많이 돌려야해서 비효율적 (그래서 AES는 파이스텔 안 씀)

### DES

| 항목 | 값 |
|---|---|
| 회전 수 | 16회 |
| 블록 | 64비트 |
| 키 | 56비트 |
| 구조 | 파이스텔 |

S-Box: 6비트→4비트 매핑, 8개 존재 → 혼돈 제공.

### 이중/삼중 DES

56비트 키는 하루면 뚫리기에, 확장 시도.

- 이중 DES: 112비트 효과지만 단독 DES보다 안전하지 않음
- 삼중 DES: 키 2개, EDE(암호화-복호화-암호화) 방식
  - 키 2개면 충분, 3개 써도 보안 향상 없음
  - EDE 이유: 하위 호환성

### AES

- 블록 암호지만 파이스텔 아님
- 고도의 수학적 구조, 혼돈+확산 모두 사용
- 4가지 연산: SubBytes(혼돈), ShiftRows(확산), MixColumns(확산), AddRoundKey(키 적용)
ㄴ 모두 역함수 존재

### TEA

- 블록 64비트, 키 128비트, 모든 연산 mod 2³²
- 아주 단순한 F → 대신 회전 수 많이
- IoT·임베디드용, 실서비스엔 안 씀
- 키 공격은 무시해도 될 만큼 성공 확률 낮음 

### 블록 암호 모드

| 모드 | 방식 | 특징 |
|---|---|---|
| ECB | 블록 독립 암호화 | 빠름·병렬 O, 같은 평문→같은 암호문 → 패턴 노출 |
| CBC | 이전 암호문과 XOR 후 암호화 (첫 블록은 IV) | 패턴 숨김, 직렬 처리, 오류 전파 |
| CTR | 카운터 암호화해 키스트림 사용 | 임의 접근 가능, 병렬 O |

CBC는 동일한 평문이 동일한 암호문을 생성하지 않음

### 블록 암호의 용도

- 불안정한 채널로 전송되는 데이터 보호
- 불안정한 저장 매체의 데이터 보호

---

## 3.4 무결성

- **기밀성**: 비인가자의 읽기 방지
- **무결성**: 비인가자의 쓰기 탐지

### MAC (Message Authentication Code)

메시지 변조 여부 + 발신자 인증을 보장하는 암호화된 체크섬.

> C₀ = E(P₀ ⊕ IV, K), C₁ = E(P₁ ⊕ C₀, K), ..., C_{N-1} = **MAC**

- 송·수신자가 비밀키 K 공유
- 수신자가 MAC 재계산 → 일치하면 무결성+인증 OK

# 4주차 - 공개키 암호

## 4.1 개요

공개키 암호는 암호화와 복호화에 서로 다른 두 개의 키를 사용. 비대칭 암호라고도 함.

- 공개키: 누구나 아는 키 (암호화용)
- 개인키: 본인만 아는 키 (복호화용)
- 기반: 트랩도어 단방향 함수 (한쪽은 쉬운데 반대로 되돌리기는 어려움)
- 평문 표기는 M (대칭키에서는 P/C)

### 디지털 서명

- 자필 서명과 같은 역할
- 개인키로 서명 → 공개키로 검증
- 본인만 서명 가능, 누구나 확인 가능

### 주요 공개키 암호체계

배낭암호, RSA, 디피-헬먼, ECC

---

## 4.2 배낭암호 
NP-완비 문제인 배낭문제를 기반으로 함.

### 배낭문제란?

무게 배열에서 일부를 골라 합이 목표값 S가 되게 하는 0/1 조합을 찾는 문제.

> 예) W = (85, 13, 9, 7, 47, 27, 99, 86), S = 172
> → 85+13+47+27 = 172 → a = (1,1,0,0,1,1,0,0)

### 키 생성 절차

1. 슈퍼증가 배낭 생성 (각 요소가 이전 요소들의 합보다 큼 → 풀기 쉬움)
2. 승수 m, 계수 n으로 일반 배낭으로 변환 (풀기 어려움)
3. 공개키 = 일반 배낭
4. 개인키 = 슈퍼증가 배낭 + 변환값(m⁻¹)

> 일반 배낭으로 암호화는 쉽고, 개인키 없이 복호화는 어려움

### 예시

- 개인키: (2, 3, 7, 14, 30, 57, 120, 251), m=41, n=491
- 공개키: (82, 123, 287, 83, 248, 373, 10, 471)
- 메시지 M=10010110 → C = 82+83+373+10 = 548
- 복호화: 12·548 mod 491 = 193 → 슈퍼증가 배낭으로 쉽게 풀이

---

## 4.3 RSA

Rivest, Shamir, Adleman 이름에서 유래. **소인수분해 문제** 기반.

### 키 생성

1. 큰 소수 p, q 선택
2. N = pq
3. (p-1)(q-1)과 서로소인 e 선택
4. ed ≡ 1 (mod (p-1)(q-1)) 만족하는 d 계산

> 공개키: (N, e) / 개인키: d

### 암·복호화

> 암호화: C = Mᵉ mod N
> 복호화: M = Cᵈ mod N

### 예시

- p=11, q=3 → N=33, (p-1)(q-1)=20
- e=3 → d=7 (3·7=21≡1 mod 20)
- 공개키 (33,3), 개인키 7
- M=15 암호화: C = 15³ mod 33 = 9
- 복호화: M = 9⁷ mod 33 = 15 

### 제곱의 반복 (속도 최적화)

큰 수 지수 계산을 빠르게 하는 법. 지수를 이진수로 보고 고차 비트부터 한 비트씩 처리.

> 예) 5²⁰ → 20 = 10100₂ → (0,1,2,5,10,20) 단계로 계산

### RSA 속도 증가

- 모든 사용자가 **같은 e 사용**하면 빨라짐
- e=3: 간단하지만 **세제곱근 공격** 취약 (M이 작거나 같은 M을 3명에게 보내면 중국인의 나머지 정리로 복구 가능)
- e=2¹⁶+1: 더 안전 (공격하려면 2¹⁶+1명에게 같은 메시지 보내야 함)

## 4.4 디피-헬먼 (DH)

키 교환 전용 알고리즘. 암호화/서명 아님. 이산 로그 문제 기반.
서로 만난 적 없는 두 사람이 "공유 비밀키"를 안전하게 만들어내는 방법.

### 동작 방식

공개 정보: 소수 p, 생성자 g

1. 앨리스: 비밀 a 선택 → **gᵃ mod p**를 밥에게 전송
2. 밥: 비밀 b 선택 → **gᵇ mod p**를 앨리스에게 전송
3. 앨리스: (gᵇ)ᵃ = **gᵃᵇ mod p**
4. 밥: (gᵃ)ᵇ = **gᵃᵇ mod p**

> gᵃᵇ가 공유 비밀키가 됨. 트루디는 gᵃ, gᵇ만 봐서는 gᵃᵇ 계산 불가 (gᵃ·gᵇ=gᵃ⁺ᵇ≠gᵃᵇ)

### 중간자 공격 (MiM)

트루디가 앨리스-밥 사이에 끼어들어 양쪽과 각각 키 교환 → 메시지 가로채기 가능.

방어법
- 공유 대칭키로 DH 교환 암호화
- 공개키로 DH 교환 암호화
- 개인키로 DH 값 서명

---

## 4.5 타원곡선 암호 (ECC)

같은 보안 수준을 더 적은 비트로 달성. 단, 연산 비용은 큼.

### 타원곡선

> E: y² = x³ + ax + b (mod p)

곡선상의 점 덧셈 연산만 사용. ECC의 모든 연산은 점 덧셈.

### ECC 디피-헬먼

일반 DH의 gᵏ 대신 점의 스칼라 곱 kP 사용.

- 공개: 곡선 E, 기저점 P
- 앨리스: 비밀 A → A·P 전송
- 밥: 비밀 B → B·P 전송
- 공유 비밀: (AB)P = (BA)P

---

## 4.6 공개키 표기법

| 표기 | 의미 |
|---|---|
| `{M}앨리스` | 앨리스의 공개키로 M 암호화 |
| `[C]앨리스` | 앨리스의 개인키로 C 복호화 |
| `[M]앨리스` | 앨리스의 개인키로 M에 서명 |

- 중괄호 { } = 공개키 연산
- 대괄호 [ ] = 개인키 연산
- 공개키·개인키는 서로 역연산: `[{M}앨리스]앨리스 = {[M]앨리스}앨리스 = M`
