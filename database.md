# 프로시저

``` sql
CREATE PROCEDURE [프로시져명] ([@파라미터])
AS
BEGIN
    [쿼리명]
END
```

Book 테이블에 한 개의 튜플을 삽입하는 프로시저를 작성하시오.
```sql
USE madang 
DROP PROCEDURE InsertBook
GO -- 문장을 끊는 표시

CREATE PROCEDURE InsertBook(@BookID INT, @BookName NVARCHAR(40), @Publisher NVARCHAR(40), @Price INT)
AS 
BEGIN
    INSERT INTO dbo.Book(bookid, bookname, publisher, price)
    VALUES (@BookID, @BookName, @Publisher, @Price);
END
GO

EXEC InsertBook 12, "스포츠과학", "마당과학서적", 25000
SELECT * FROM Book;
GO
```

Book 테이블에 저장된 도서의 평균 가격을 반환하는 프로시저를 작성하시오
``` sql
USE madang
DROP PROCEDURE AveragePrice
GO

CREATE PROCEDURE AveragePrice(@AverageVal INT OUTPUT)
AS
BEGIN
	SET @AverageVal = (SELECT AVG(price) 
						FROM Book
						WHERE price IS NOT NULL);
END
GO

DECLARE @Avg INT
EXEC AveragePrice @AverageVal = @Avg OUTPUT
SELECT @Avg '책값 평균'
GO
```

# 트리거

트리거는 테이블에서 특정 이벤트(INSERT/UPDATE/DELETE)가 발생할 때 자동 실행되는 프로시저다. (ex. Book이 삭제될 때 로그를 자동으로 남기고 싶다.)

``` sql
CREATE TRIGGER 트리거명 ON 테이블명 [AFTER INSERT, UPDATE, DELETE] 
AS
BEGIN
    -- 트리거 내용
END;
```

*직사각형 - 강한 개체 타입*
- 자기 혼자서도 존재할 수 있는 개체
- 기본키(PK)를 스스로 가진다
- EX) 학생, 상품, 고객

*이중 직사각형 - 약한 개체 타입*
- 혼자서는 구별될 수 없는 개체
- 고유한 기본키가 없음, 다른 개체의 PK를 사용해야 구별됨
- EX) 수강(학생 + 과목)

*동그라미 - 단순 속성*
- 더 이상 분해 불가능한 속성

*동그라미 - 복합 속성*
- 독립적인 의미를 가진 속성으로 분해할 수 있는 속성
-         주소
       시      동

*동그라미 - 단일값 속성*
- 값이 단일임

*이중 동글라미 - 다중값 속성*
- 값이 다중값임
- 학위(학사, 석사)

*동그라미 - 저장 속성*
- 다른 속성의 영향 없이 단독으로 저장되는 속성

*동그라미 - 유도 속성*
- 다른 저장 속성으로부터 유도된 속성
- 생년월일 - 나이

*마름모 - 관계 타입*

## 이상현상

잘못된 데이터베이스 설계로 생기는 현상인듯

1. 삭제이상
: 원래 지울 필요 없는 중요한 정보까지 함께 삭제되는 문제.

2. 삽입이상
: 중복 구조 때문에 추가할 수 없거나, 불필요한 데이터를 같이 넣어야 하는 문제.

3. 수정이상
: 하나만 수정해도 나머지는 그대로 남아 데이터 불일치가 생기는 문제.

## 정규화

릴레이션을 분해해서 이상현상을 없애는 과정

### 제 1정규형
속성값이 유일하도록 바꾸는 것

EX) 고객취미들(이름, 취미들) -> 고객취미(이름, 취미)

### 제 2정규형
기본키가 복합키일 때, 기본키의 일부만으로 일반 속성이 결정되지 않는 것

EX) | 학번 | 과목번호 | 학생이름 | 과목명 성적 |

기본키: (학번, 과목번호)

기본키의 일부가 일부 속성을 결정지을 수 있기때문에 2NF 위반

### 제 3정규형
제 2정규형을 만족하고, 기본키가 아닌 속성 간의 이행적 함수적 종속이 없는 관계

> 이행적 함수적 종속을 한 문장으로
>> A 때문에 B가 정해지고,
>> B 때문에 C가 정해지면,
>> A 때문에 C도 정해지는 것 

### BCNF 정규형
모든 결정자가 후보키인 관계
다른 값을 결정하는 애는 반드시 키여야 한다.

# 8장

DBMS에서 데이터를 다루는 논리적인 작업의 단위

- 원자성 : 트랜젝션에 포함된 작업은 전부 수행되거나 전부 수행되지 않아야함
- 일관성 : 트랜잭션을 수행하기 전이나 수행한 후 데이터베이스는 항상 일관된 상태를 유지해야함
- 고립성 : 수행 중인 트랜잭션에 다른 트랜젝션이 낄 수 없음
- 지속성 : 변경한 데이터를 여구히 저장해야함

## 제어 명령어

```sql
BEGIN --트랜잭션 시작, START TRANSACTION도 가능
COMMIT --트랜잭션 종료
ROLLBACK --트랜잭션을 전체 혹은 <SAVEPOINT>까지 무효화 시킴
SAVE --<SAVEPOINT>를 만듬
```

## 갱신손실 문제
두 개의 트랜잭션이 한 개의 데이터를 동시에 갱신할 때 발생하는 문제


## 락
갱신손실 문제를 해결하기 위해 사용함
데이터를 읽거나 수정할 때 데이터에 표시하는 잠금장치

대기상태로 만들기 때문에 응답시간에 영향을 줄 수 있음.

읽기를 할 때 사용하는 락: 공유락(LS)
읽고 쓰기를 할 때 사용하는 락: 배타락(LX)

### 2단계 락
1단계: 잠그기만 하는 시간
2단계: 열기만 하는 시간
**단, 열고 다시 닫으면 안됨**

### 데드락
둘 이상의 트랜잭션이 서로 상대방이 가진 자원을 기다리며 무한 대기 상태에 빠지는 현상.

*해결방법?* : 그냥 작업 중 하나를 강제로 중지시킴

### 오손 읽기
커밋 안 된 데이터를 읽는 것

> 언제 발생하냐?
- 여러 사람이 동시에 작업할 때
- 락이 제대로 걸리지 않았을 때
- 낮은 격리 수준에서

### 반복불가능 읽기
한 트랜잭션이 동일한 데이터를 여러 번 읽는 동안
다른 트랜잭션이 해당 데이터를 수정하여 읽기 결과가 달라지는 현상.

### 유령데이터 읽기
아까는 없던 데이터가, 다시 보니까 갑자기 생겨 있는 현상

## 트랜잭션 고립 수준 명령어

|  | 오손 읽기 | 반복불가능 읽기 | 유령데이터 읽기 |
| --- | --- | --- | --- |
| READ UNCOMMITTED  | 가능 | 가능 | 가능 |
| READ COMMITTED | 불가능 | 가능 | 가능 |
| REPEATABLE READ | 불가능 | 불가능 | 가능 |
| SERIALIZABLE | 불가능 | 불가능 | 불가능 |


## 회복

장애 유형
- 시스템 충돌
- 미디어 장애
- 응용 소프트웨어 오류
- 자연재해
- 부주의 혹은 태업

### 로그 파일
```SQL
<트랙잭션번호, 로그 타입, 데이터 항목 이름, 수정 전 값, 수정 후 값>
```

```sql
--예시
<T1, START>
<T1, UPDATE, Customer (박지성).balance, 100000, 90000>
<T1, UPDATE, Customer (김연아).balance, 100000, 110000>
<T1, COMMIT>
```

>트랜잭션의 재실행(REDO)

장애 후 시스템을 재가동 했을 때, 로그파일에 트랜잭션의 시작, 종료가 있는 경우

>트랜잭션의 취소(UNDO)

장애 후 시스템을 재가동 했을 때, 로그파일에 트랜잭션의 시작만 있는 경우

>즉시 갱신(Immediate Update)

👉 작업 중에도 바로바로 DB에 써버림

>지연 갱신(Deferred Update)

👉 작업 끝날 때까지 메모장에만 적어두고, 마지막에 한 번에 씀

>✅체크포인트

“여기까지는 안전하다!”라고 표시해 두는 중간 저장 지점

체크포인트 전 COMMIT 기록이 있는 트랜잭션
- 아무작업 필요 없음

체크포인트 후 COMMIT 기록이 있는 트랜잭션
- REDO를 진행(즉시갱신, 지연갱신)

체크포인트 후 COMMIT 기록이 없는 트랜잭션
- UNDO를 진행(즉시 갱신)
- 아무작업 필요없음(지연 갱신)
